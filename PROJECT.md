## 2. Бізнес-контекст

- **Свідомий вибір не масштабуватись.** Пробувала другий салон у центрі Варшави — закрила, бо "забагато сил і нервів". Самозайнятість — її цільова модель.
- **Конкуренція поряд:** новий конкурент відкрився **150 м від салону** ~весна 2026. Сучасний клієнтський додаток — частина її стратегії диференціації.
- **Канали залучення нових клієнтів:** Google Maps (основне, відгуки!), Facebook, Instagram (дуже мало в PL). WhatsApp популярніший за Telegram.
- **Конверсія дзвінків:** дуже висока, no-shows — рідкісні. SMS-нагадування за день до візиту вже працює і клієнтам подобається.

## 3. Що ми будуємо і ЧОМУ

**НЕ Booksy і не клон CRM.** Мета — **внутрішній CRM салону + клієнтський кабінет як premium-feature**, бо:

1. У Iванки був досвід з Booksy у варшавському салоні — клієнти скасовували онлайн за 5 хв до візиту, передплата зменшувала кількість записів. Тому **онлайн-бронювання повністю виключене**.
2. Польські салони такого не пропонують ("я не зустрічала такого"). Це конкурентна перевага як сервіс.
3. Iванка хоче замінити паперовий зошит + спроби в Google Calendar (кинули за 2 місяці) на єдину базу зі швидким пошуком.

### Цілі продукту (її словами)

- Полегшити внутрішню роботу: одна база замість зошитів і Google-календаря
- Швидкий доступ до історії клієнта: ціна, фото до/після, нотатки, остання стрижка
- Картки лояльності в цифрі (5 стрижок → 10% знижка, **залежить від породи** — див. `breed_eligible_for_loyalty`)
- Сучасний "клієнтський кабінет" — диференціація на польському ринку

### Що НЕ робимо (явно відкинуто)

- Онлайн-бронювання, скасування, перенесення (тільки телефон / SMS)
- Передплата при записі
- Українська локалізація для клієнтів (тільки PL)
- Публічні відгуки про грумерів (між собою або зовні)
- Складна анкета оцінки візиту (просто 1–5 зірочок + опційний текст)
- Telegram Mini App (поляки не сидять у Telegram) — мігруємо у звичайний web-додаток за посиланням + QR-код біля рецепції

## 4. Ключові правила і ризики

- **Аналітика по дохідності — лише в Iванки.** Iнші грумери бачать тільки власні цифри. Причина: "якщо хтось бачить цифри, думає що я кокоси грібую". Грумери не вирахують оренду / податки / витрати.
- **Не давати клієнту змогу самостійно скасовувати** — буде Booksy-сценарій (скасування за 5 хв).
- **Чітко розділити "інформація для клієнта" і "внутрішні нотатки"** у groomer view. Iванка прямо просила: подвійне підтвердження "Це побачить клієнт", виділення кольором/рамкою. Боїться, що грумер випадково напише туди не те.
- **Внутрішньої конкуренції між грумерами немає** — усі бачать весь розклад, усі можуть писати/редагувати будь-який запис. (У варшавському салоні було інакше — там була конкуренція. Тут — ні.)
- **Формальний тон у комунікації з власниками собак.** Iванка: "Чим більше ми стираємо кордони, тим більше клієнти сідають на голову". До собаки можна сюсюкати, до власника — "Dzień dobry", не "Cześć".

## 5. Шлях користувача

### Клієнт (власник собаки)

1. **Перший раз:** заходить через QR-код біля рецепції або посилання в шапці Instagram → онбординг-анкета (порода з dropdown, телефон, кличка, день народження, фото, алергії, парфуми так/ні, смаколики так/ні, особливості здоров'я) → consent-галочки (SMS, регламент, RODO) → потрапляє у свій кабінет.
2. **Повтор:** вводить телефон → отримує SMS-код → у кабінеті бачить картку собаки, наступний візит (виділено акцентом), карту лояльності, історію з фото до/після і нотатками від грумера.
3. **Запис на стрижку:** **тільки по телефону.** В кабінеті — кнопка "Zadzwoń" / "Napisz SMS". Самостійного запису немає.
4. **Після візиту:** при наступному заході — fullscreen-вікно "Оцініть останній візит" (1–5 зірочок + опційний текст). Можна закрити, повториться наступного разу.
5. **Поза app:** отримує SMS за день до візиту з нагадуванням; SMS у день народження собаки; SMS-нагадування "час на стрижку" через N тижнів.

### Грумер (4 акаунти: Iванка, Оля, Мартін, помічниця)

1. Заходить → дашборд: усі клієнти з чіпами (день народження, алергії, давно не був). Пошук по імені і по телефону.
2. Розклад: день / тиждень. Усі грумери бачать графіки всіх. Записи фарбуються по грумеру.
3. Створення візиту: дата/час, грумер (dropdown), декілька послуг (чекбокси: повний пакет + extras), опис стрижки, ціна (можна перевизначити), нотатки внутрішні (червона рамка), нотатки для клієнта (зелена рамка з подвійним підтвердженням), фото до/після.
4. У карті клієнта — два візуально розділені блоки: "Інформація для власника" і "Внутрішні нотатки салону". В історії візитів — фото до/після відкриваються.
5. **Iванка (тільки вона):** аналітика з розбивкою по грумерах (дохід / клієнти / рейтинг). API захищене ролевою перевіркою.
6. **Створення клієнта вручну** — для людей старшого віку, які не заповнять самі.

## 6. Технічний стек і інфраструктура

**Поточний:**

- FastAPI ([demo.py](demo.py)) + aiogram (Telegram-бот) + APScheduler (щоденні нагадування о 09:00)
- SQLite ([salon.db](salon.db)) — для демо; перед прод-запуском мігруємо на Postgres
- Jinja2 templates + vanilla JS SPA (`static/app.js`, `static/app.css`) — мобайл-перш
- Деплой: Render (`render.yaml`), Procfile, runtime.txt

**План змін під прод:**

- Прибрати aiogram → додати SMS-провайдера (кандидати: SMSAPI.pl, Twilio, SerwerSMS). Перевірити можливість sender ID з назвою салону — у Iванки фірма зареєстрована на іншій назві, треба підтвердження документами.
- Postgres + asyncpg
- Фото — наразі base64 у БД; перенести на S3 / Cloudflare R2
- Авторизація: SMS-OTP для клієнтів; PIN / token для грумерів з полем `role` (owner_salon vs groomer)
